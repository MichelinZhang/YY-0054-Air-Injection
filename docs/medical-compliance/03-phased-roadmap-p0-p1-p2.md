# YY-0054 标准化改造路线图（P0 / P1 / P2）

## 1. 目标与原则
- 目标：以 IEC 62304 + ISO 14971 + ISO 13485 为主线，将当前系统从“可用原型”推进到“可审计、可验证、可发布”的医疗软件工程状态。
- 实施原则：
  - 先控制风险，再补齐体系。
  - 每阶段必须有可验收交付物。
  - 文档与代码同步演进，防止“实现先行、证据缺失”。

## 2. 总体阶段划分
- P0（安全与正确性底线）：先消除高风险运行缺陷与误用路径。
- P1（工程化与可追溯）：建立验证闭环和配置/依赖受控。
- P2（体系化合规）：完成组织级流程与发布治理对齐。

## 3. P0：安全与正确性底线

### 3.1 目标
- 保障运行链路可控，不允许“无权限、无审计、数据不一致”继续进入结果输出。

### 3.2 任务包
- 访问控制与角色最小化
  - 为 REST 与 WebSocket 增加认证机制与角色校验。
  - 默认关闭 `allow_origins=["*"]` 的开放策略，按环境配置白名单。
- Mock 生产隔离
  - 生产环境禁止 `force_mock` 与自动 mock 回退进入保存流程。
  - mock 模式必须带有强提示与审计事件。
- 结果完整性控制
  - 保存前校验“测量记录 camera_id 与最新帧一致性”。
  - 结果文件改为原子写并增加损坏检测。
- 错误治理
  - 统一错误码规范与错误响应结构。
  - 对外错误信息最小披露，对内保留诊断上下文。
- 审计日志最小集
  - 记录会话启停、模式切换、测量、保存、参数修改、异常。

### 3.3 验收标准
- 未认证用户无法调用受保护 API/WebSocket。
- 生产配置下 mock 流程不可保存结果。
- 任一测量记录缺失对应帧时，保存必须失败且返回规范错误码。
- 审计日志可追溯关键操作，且可按 `session_id/record_id` 检索。
- 全部新增控制项有自动化测试（单元+集成）。

### 3.4 建议里程碑
- M1：认证授权与 CORS 收敛完成。
- M2：mock 隔离与结果完整性校验完成。
- M3：错误码与审计日志完成并通过回归测试。

## 4. P1：工程化与可追溯

### 4.1 目标
- 形成“需求-风险-测试-实现”闭环，确保变更可评估、可回归、可解释。

### 4.2 任务包
- 追溯矩阵 RTM 建立
  - 需求条目（功能与安全）映射到风险控制、测试用例、代码模块。
- SOUP 管理
  - 建立海康 SDK、OpenCV、NumPy、FastAPI 等 SOUP 台账。
  - 定义版本冻结、升级评估、回归验证流程。
- 算法回归基线
  - 为 `reading_assist_service` 建立受控数据集与回归指标。
  - 增加阈值漂移检测与发布阻断门槛。
- 配置与环境分级
  - 区分开发/验证/生产配置，形成受控配置基线。

### 4.3 验收标准
- 每个高风险需求至少有一条风险控制与一条验证用例关联。
- SOUP 台账完整且能定位当前发布使用的版本与验证记录。
- 算法回归报告可自动生成，低于阈值时 CI 阻断发布。
- 环境分级配置可复现，且发布包配置不可被未授权修改。

### 4.4 建议里程碑
- M4：RTM 初版 + SOUP 台账初版。
- M5：算法基线与回归框架上线。
- M6：环境分级与发布前检查清单上线。

## 5. P2：体系化合规

### 5.1 目标
- 从项目内控制扩展到质量体系与发布流程控制，满足审计和注册资料沉淀需求。

### 5.2 任务包
- 生命周期文档体系
  - SDP、SRS、SAD、SVP/SVR、风险管理文件、维护计划等模板化并受控。
- 变更与问题管理
  - 缺陷分级、CAPA、偏差处理、回归影响分析流程固化。
- 发布与运维治理
  - 版本发布记录、回滚策略、日志保留策略、监控指标标准化。
- 文档一致性机制
  - API、README、测试、用户文档一致性检查纳入流水线。

### 5.3 验收标准
- 关键生命周期文档齐备并可追溯到当前版本。
- 每次发布均有完整审查记录（变更、风险、验证、审批）。
- 运维侧具备事件复盘所需日志与指标证据。
- 文档一致性检查可自动执行并阻断不一致发布。

### 5.4 建议里程碑
- M7：生命周期文档模板与流程固化。
- M8：发布审查与运维治理闭环上线。
- M9：内部模拟审计通过。

## 6. 执行依赖与组织建议
- 角色建议：
  - 软件负责人：路线图推进与跨团队协调。
  - 算法负责人：测量语义、基线数据、性能阈值。
  - 测试/质量负责人：追溯矩阵、验证策略、发布门禁。
- 依赖建议：
  - 先落实 Intended Use 与软件安全分类，再细化控制深度。
  - 与硬件/法规团队同步确认“可用于判定”边界和声明口径。

## 7. 推荐的首轮落地顺序
1. 执行 P0-M1（认证授权 + CORS 收敛）。
2. 执行 P0-M2（mock 隔离 + 保存完整性）。
3. 执行 P0-M3（错误码 + 审计日志 + 测试）。
4. 进入 P1（RTM、SOUP、算法基线）。
5. 在 P1 稳定后推进 P2 文档与流程体系化。
